<objective>
Three surgical fixes for the Settings → Integrations + Calendar surfaces:

1. **Calendar week view fetches and renders events.** Root cause: `src/main/integrations/google/sync-calendar.ts` (a) INSERTs into `calendar_event` WITHOUT `provider_key` and `account_id`, but the read-path IPC `aria:calendar:list-events-range` (`src/main/ipc/calendar.ts:289-344`) JOINs to `provider_account` and filters by `e.provider_key IS NOT NULL AND e.account_id IS NOT NULL` — so every Google-synced row is invisible; (b) reads/writes the dropped legacy `calendar_account` table (`SELECT email, sync_token FROM calendar_account WHERE id = 1`), which throws and is silently caught by `runTick`'s try/catch, so the tick never completes and no rows are ever written. Microsoft's `src/main/integrations/microsoft/sync-calendar.ts` is the correct shape and is the reference. Fix by lifting Google sync onto `provider_account` + `provider_sync_state` (matching MEMORY `migration_014_lift_incomplete` as the explicit followup).

2. **Remove the orphaned legacy boxes on Integrations.** The legacy `GmailRow` / `CalendarRow` / `TodoistRow` already gate themselves on `hideWhenHealthy = hasHealthyAccount(accounts, providerKey)` (`status === 'ok' && !lastError`). The orphan reappears when an account is present but has `lastError` set or `status === 'degraded'` — even though the AccountRow already surfaces both. Change `hideWhenHealthy` to PRESENCE-based: hide whenever ANY AccountRow of that provider is mounted, regardless of status. AccountRow already renders provider, email, status chip, and lastError — the legacy rows are duplicative when an account exists.

3. **Add Todoist to the Add-account modal.** Existing modal in `src/renderer/components/AddAccountModal.tsx` hard-codes a two-entry `PROVIDERS` const (microsoft + google) and a `connect()` that branches on `selected === 'microsoft' ? microsoftConnect() : gmailConnect()`. On success, call `onConnected()` (refreshes accounts) and close. Reuses every existing IPC; no new channels.
