    "# Development Guide"

    "## Prerequisites"
    List exact required tools:
    - Node.js 20 LTS (use nvm or fnm — the project is pinned to Node 20 in @types/node)
    - pnpm (not npm — the project uses pnpm; install via "npm install -g pnpm" or corepack enable)
    - Electron 41.6.1 (pinned — do NOT upgrade; see ABI section below)
    - Ollama (https://ollama.ai) — required for local LLM routing and RAG embeddings

    "## Ollama Setup"
    After installing Ollama, pull the required models:
    - "ollama pull llama3.1:8b" — sensitivity routing and local LLM fallback (Q4_K_M recommended)
    - "ollama pull nomic-embed-text" — RAG embeddings (274MB)
    Note: Ollama must be running at http://localhost:11434 before launching Aria. Aria gracefully degrades if Ollama is absent (frontier-only mode) but RAG and sensitivity routing will not work.

    "## Clone and Install"
    Three commands: git clone, cd aria, pnpm install. Note: the postinstall script (scripts/postinstall.mjs) runs automatically after install.

    "## Native Binary Rebuild (important)"
    Aria uses better-sqlite3-multiple-ciphers which is a native Node addon. It must be compiled for the correct ABI before running.
    - To run Aria desktop (Electron context): "pnpm run rebuild:native" — this builds the native binary for both Electron 41 and Node 20 (dual build).
    - To run tests (Node context): "pnpm run rebuild:native:node" — builds for Node only.
    - Why this matters: Electron uses a different V8 ABI than Node. Running the app after a "pnpm run rebuild:native:electron" will break vitest, and vice versa. The dual build (rebuild:native) handles both.
    - Warning: "pnpm install" does NOT automatically rebuild native binaries. You must run rebuild:native manually after install or after switching between app and test contexts.

    "## OAuth Credentials (bring your own)"
    Aria requires you to supply your own OAuth app credentials. The app will not connect to Google or Microsoft without them.

    Sub-section "### Google (Gmail + Calendar)":
    1. Create a project at https://console.cloud.google.com
    2. Enable the Gmail API and Google Calendar API
    3. Create OAuth 2.0 credentials (Desktop app type)
    4. Add http://localhost as an authorized redirect URI
    5. Set GOOGLE_OAUTH_CLIENT_ID and GOOGLE_OAUTH_CLIENT_SECRET in .env.local

    Sub-section "### Microsoft (Outlook + Calendar via Graph)":
    1. Register an app at https://portal.azure.com → Azure Active Directory → App registrations
    2. Add redirect URI: http://localhost (Mobile and desktop application platform)
    3. Grant delegated permissions: Mail.Read, Mail.Send, Calendars.ReadWrite, offline_access
    4. Set MICROSOFT_OAUTH_CLIENT_ID in .env.local (MICROSOFT_OAUTH_TENANT defaults to "common")

    Sub-section "### .env.local":
    Copy .env.example to .env.local and fill in your values. This file is gitignored. Production builds inject these via the electron-vite define plugin (not at runtime from the filesystem).

    "## Running the App"
    - "pnpm dev" — starts electron-vite dev server + Electron in development mode (hot reload for renderer)
    - First launch: the onboarding wizard will prompt for a vault passphrase and guide you through account setup

    "## Running Tests"
    - Run "pnpm run rebuild:native:node" first (close the Aria desktop app — it holds the better-sqlite3 native binary lock and will cause EBUSY errors)
    - "pnpm run test:unit" — runs lint:guard ratchets + vitest unit/integration suite
    - "pnpm run test:e2e" — Playwright E2E (requires a built app: "pnpm build" first)
    - "pnpm run typecheck" — TypeScript type check (electron-vite's esbuild skips type checking during build; always run this after editing main/ or preload/)

    "## Lint Guard Ratchets"
    The project uses custom grep-based static analysis in scripts/grep-*.mjs, wired into "pnpm run lint:guard":
    - grep-insight-prose-no-raw.mjs — prevents raw data from leaking into insight prose without LLM routing
    - grep-migration-callsite.mjs — ensures runMigrations is only called from the single boot path
    - grep-no-fixture-leak.mjs — ensures the force-fail migration fixture is never embedded in production code
    - grep-no-network-from-signals.mjs — prevents learning signals from making outbound network calls
    - verify-audit-view.mjs — verifies the action_audit_log SQL VIEW column set matches the expected base tables
    Run "pnpm run lint:guard" before committing. It runs automatically as part of "pnpm run test:unit".

    "## Troubleshooting"

    Sub-section "### better-sqlite3 ABI lock":
    Symptom: vitest fails with "NODE_MODULE_VERSION 141 vs 145" or EBUSY. Cause: the Aria desktop app is running and holds the native binary compiled for Electron's ABI. Fix: close the Aria app, then "pnpm run rebuild:native:node", then re-run tests.

    Sub-section "### Electron ABI pin":
    Electron is pinned at 41.6.1. Do not upgrade without also running "pnpm run rebuild:native" and verifying tests pass. The pin is tracked in devDependencies with an exact version (no caret).

    Sub-section "### esbuild skips TypeScript errors":
    electron-vite uses esbuild for compilation, which does NOT run type checking. Type errors in src/main/ or src/preload/ will silently compile and crash at runtime. Always run "pnpm run typecheck" after editing those directories.

    "## Building for Production"
    - "pnpm run build" — builds the app bundle into out/
    - "pnpm run release" — builds + publishes to GitHub Releases (requires GH_TOKEN)
    - See docs/RELEASE-RUNBOOK.md for the full release procedure including macOS notarization and Windows signing.

    "# Architecture"
    Opening sentence: "Aria follows the standard Electron process model with a strict separation between main (Node.js), preload (context bridge), and renderer (React)."

    "## Process Model"
    Explain the three-process split:
    - Main process (src/main/index.ts): Node.js, owns all data access (SQLCipher DB), LLM calls, OAuth flows, file I/O, scheduling. Never exposed directly to the renderer.
    - Preload script (src/preload/index.ts): thin contextBridge layer. Exposes a typed `window.api` object to the renderer — one `invoke(channel, ...args)` function. No business logic here.
    - Renderer (src/renderer/): React 18 SPA. All IPC calls go through window.api.invoke(). No direct Node.js access.

    Include a compact ASCII diagram:
    ```
    Renderer (React)
        │ window.api.invoke(channel, args)
    Preload (contextBridge)
        │ ipcRenderer.invoke()
    Main Process (Node.js)
        ├── IPC handlers (src/main/ipc/)
        ├── SQLCipher DB (src/main/db/)
        ├── Integrations (src/main/integrations/)
        ├── LLM Router (src/main/llm/)
        └── Background scheduling (src/main/briefing/, src/main/learning/, etc.)
    ```

    "## IPC Surface"
    The IPC layer (src/main/ipc/index.ts) registers all handler modules at boot. Each module (approvals.ts, briefing.ts, gmail.ts, calendar.ts, microsoft.ts, ask.ts, etc.) registers its channels with ipcMain.handle(). The preload exposes a single invoke() bridge — the renderer does not know channel strings, only the typed API surface.

    "## Database and Migrations"
    - SQLite database using better-sqlite3-multiple-ciphers (SQLCipher, AES-256 whole-database encryption).
    - Key stored in OS keychain via Electron safeStorage; never written to disk as plaintext.
    - Migrations live in src/main/db/migrations/ — numbered SQL files embedded at build time in embedded.ts and run sequentially on every boot via runner.ts.
    - The migration runner is called from a single boot-time callsite (enforced by scripts/grep-migration-callsite.mjs).

    "## Provider Adapters"
    All external data integrations live under src/main/integrations/:
    - google/ — Gmail and Google Calendar via googleapis + google-auth-library (OAuth loopback IP flow)
    - microsoft/ — Outlook and Graph Calendar via @azure/msal-node + microsoft-graph-client (PKCE)
    - todoist/ — Todoist task sync
    - registry.ts — ProviderRegistry maps accountId to adapter instances
    - sync-orchestrator.ts — coordinates polling schedules across providers via p-queue

    "## Sensitivity Router and LLM Routing"
    - src/main/llm/sensitivityClassifier.ts: two-stage classifier (regex prefilter → local Ollama generateObject via zod schema) that assigns a sensitivity level to each piece of content.
    - src/main/llm/router.ts: routes LLM calls to LOCAL (Ollama) or FRONTIER (Anthropic/OpenAI/Google) based on sensitivity classification. All routing decisions written to routing_log table.
    - Local model: Llama 3.1 8B / Qwen 2.5 7B via Ollama (localhost:11434).
    - Frontier: Vercel AI SDK (ai package) with @ai-sdk/anthropic, @ai-sdk/openai, @ai-sdk/google.
    - PII-flagged content never reaches a frontier model.

    "## Approval Chokepoint (assertApproved)"
    All outbound actions — email sends, calendar writes, task pushes — must pass through the approval gate before executing:
    - src/main/approvals/gate.ts exports assertApproved(approvalId, db): throws if the approval row is not in 'approved' state.
    - All write paths (src/main/integrations/send.ts for email, src/main/integrations/write-event.ts for calendar) call assertApproved before any external API call.
    - The approval chokepoint is enforced by a static grep ratchet in scripts/ that runs as part of lint:guard.
    - This means Aria cannot send an email or change a calendar event without an explicit user approval action.

    "## Background Scheduling"
    - node-cron (src/main/briefing/scheduler.ts and per-module schedule files) handles daily briefing generation, integration polling, learning aggregation, and recap generation.
    - p-queue serializes all LLM calls to prevent rate-limit storms and make cost predictable.
    - Electron powerMonitor suspends cron jobs on machine sleep and resumes on wake, preventing cron storms after a laptop wakes from overnight sleep.
    - When the database is sealed (locked), cron jobs silently skip and queue a catch-up run on first unlock.

    "## Background Activity and Tray (Phase 12)"
    - Close-to-tray: src/main/background/window-decisions.ts handles the window close event, routing to hide vs quit based on user preference (closeToTray setting).
    - Tray: src/main/tray/ — icons.ts resolves platform-specific icon assets, menu.ts builds the native context menu, notify.ts sends OS notifications (briefing ready, etc.).
    - Auto-launch: src/main/background/prefs.ts mirrors the autoLaunch setting to the OS login item.
