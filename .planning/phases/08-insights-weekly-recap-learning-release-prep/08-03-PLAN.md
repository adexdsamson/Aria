    ## Per-source signal-emit boundary — DECISION TREE (one shape per source)

    **Context — why this section was re-revised:** Round 1 of this plan said "emit AFTER commit." Round 2 revision tightened that to "SAME-TRANSACTION emit" thinking it strengthened atomicity. Peer review round 2 (B-3) caught that the tighter shape is wrong: an IPC handler that performs an approval state transition AND a learning signal write inside one `db.transaction()` AND then calls an external HTTPS API (Gmail / Google Calendar / Todoist) cannot hold the SQLite write transaction open across the HTTPS call — a sqlite write lock is in-process and synchronous; an HTTPS request is asynchronous and can take seconds. Holding the txn open across the await deadlocks better-sqlite3.

    More importantly, MEMORY `project_aria_approve_silent_failure` documents the existing Phase 4 architectural followup: state transitions to 'approved' BEFORE the external API call, and on external throw the row is left in 'approved' with no Gmail/Calendar write. The fix is to roll back / re-transition state on external failure, NOT to wrap state+signal in one txn. Wrapping state+signal in one txn does NOT solve the Phase 4 problem — it creates a NEW problem: a signal that claims an action happened, written atomically with the state change, but the external write that the action depends on hasn't fired yet (and may fail).

    Restored shape (round 3 = round 1 = research §Anti-Patterns "signal capture happens AFTER user action commits, never speculatively"):

    1. **approval source (EMIT-AFTER-EXTERNAL-WRITE):**
       Ordering inside the approval IPC handler:
       ```
       db.transaction(() => transitionTo(db, approvalId, 'approved', patch))();   // 1. state COMMIT
       const result = await applyCalendarChange(db, approvalId, deps);            // 2. HTTPS call (outside txn)
       if (result.ok) {
         writeSignal(db, { source: 'approval', kind: 'approval.accept', payload });  // 3. signal AFTER success
       } else {
         // Phase 4 followup: re-transition state to 'failed' (NOT silent stay-at-approved)
         db.transaction(() => transitionTo(db, approvalId, 'failed', { last_error_message: result.error }))();
         // NO signal emitted on external failure.
       }
       ```
       Three sequential atomic operations. No txn spans the HTTPS call. Phase 4 silent-write is closed by the explicit `transitionTo('failed')` in the failure branch (NOT by this Stream 3 plan — Stream 3 just adds the signal at step 3 AND asserts in Test 5b that the signal is NOT written in the failure branch).

       If the Phase 4 chokepoint refactor has not yet shipped at execution time (separate cross-phase work), Test 5b STILL holds: it forces an external throw, observes that writeSignal is not called, and tolerates the row staying in 'approved' (a documented Phase 4 bug — Stream 3 does not regress it but does not fix it either).

    2. **recap source (EMIT-WITH-SECTION-EDIT):**
       finalizeRecap (Plan 08-02 Task 3) writes `weekly_recap_section_edit` rows. There is no external API call in the recap finalize path — it is database-only. Wrap section_edit INSERT and learning_signal INSERT in one `db.transaction()`. This is the legitimate same-txn case.

    3. **briefing source (DIRECT WRITE — the click IS the signal):**
       BRIEFING_FEEDBACK and BRIEFING_INSIGHT_DISMISS IPC handlers each open a `db.transaction()` that writes the source row (briefing_feedback or briefing_dismiss) AND a learning_signal in one shot. No external API in scope; same-txn is correct here.

    4. **qa source (DIRECT WRITE — same as briefing):**
       appendTurnFeedback wraps `UPDATE rag_turn SET thumb=?` + learning_signal INSERT in one `db.transaction()`. No external API; same-txn is correct.
