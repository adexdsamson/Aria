<objective>
Unify the Google adapter (Phases 2 + 4) and the Microsoft adapter (Plan 05-01) behind a single `Provider` interface with capability flags, and generalize the two write chokepoints (`applyCalendarChange`, `sendApprovedEmail`) to dispatch via a `ProviderRegistry`. This is the refactor pass — no new wire behavior; same Google paths continue to work end-to-end, and Outlook send + calendar write now ride the same approval/audit/ratchet machinery.

Also fixes L-04-05 (transitionTo-before-write race) as a cross-cutting architectural improvement. For calendar_change (idempotent PATCH+etag) the fix is simple ordering. For email_send (NOT naturally idempotent) the fix is the explicit 'sending' interim state machine (B2 Option A) — preventing a duplicate-send hazard if the app crashes between dispatch and approval-state update.

Closes: EMAIL-02 (send half) and CAL-02 (write half). CAL-08 canonical-conversion-at-boundary preserved.

Purpose: Single source of truth for the Provider contract eliminates the duplication CONTEXT explicitly authorized during Plan 05-01. Cross-cutting features (Phase 3 triage, Phase 4 scheduler) become provider-agnostic without per-provider branches in feature code.

<interfaces>
Provider interface (verbatim from RESEARCH Pattern + Capability Flags):
  - ProviderKey = 'google' | 'microsoft'
  - CanonicalMessage, CanonicalEvent, DeltaResult<T>
  - MailCapability: listMessagesDelta, getMessage, sendMessage(canonical, { idempotencyKey: string })
  - CalendarCapability: listEventsDelta, listEventsWindow, getEvent, patchEvent, insertEvent, eventInstances, freeBusy
  - ProviderCapabilities: recurrenceFormat, supportsSendUpdates, mailLabelModel, mailSendReturnsId
  - Provider: { providerKey, accountId, accountEmail, capabilities, mail?, calendar? }

Capability flags (Google vs Outlook divergences):
  - mail.threadIdField: 'threadId' | 'conversationId'
  - mail.draftFlow: 'oneCall' (google) | 'twoCall' (outlook — createReply+send)
  - mail.sendReturnsId: true (google) | false (outlook)
  - calendar.recurrenceFormat: 'rrule' | 'graph-pattern'
  - calendar.tombstoneShape: 'status' | 'odata-removed'
  - calendar.supportsSendUpdates: true (google) | false (outlook)

Approval state machine (after migration 012a):
  approved → sending → sent              (email happy path)
  approved → sending → failed            (email dispatch exception)
  approved → sent                        (calendar happy path; no 'sending' interim)
  approved → failed                      (calendar dispatch exception)
  sending  → needs-operator-decision     (boot recovery, B2)

Idempotency:
  - approval.idempotency_key: UUID v4 issued at insert time (migration 012a default new uuid())
  - Email dispatch passes the key:
    * Gmail: X-Aria-Idempotency-Key header in the raw RFC822 body (Gmail preserves custom headers)
    * Graph: passed as outbound message internetMessageHeaders entry { name: 'X-Aria-Idempotency-Key', value: <uuid> }
  - Provider de-dup is best-effort; the PRIMARY mechanism is the 'sending' state preventing double-dispatch from Aria itself.
