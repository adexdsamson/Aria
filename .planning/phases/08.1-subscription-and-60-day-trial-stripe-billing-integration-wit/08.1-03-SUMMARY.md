# Phase 08.1 Plan 03: Paywall UX + Trial Banner + Settings Subscription Summary

**Ship of the user-facing entitlement layer.** Plan 02 froze the IPC contract and built the chokepoint; Plan 03 surfaces it as a paywall when the gate is closed, a countdown banner when the trial is ending, and a paste-form + Settings management surface for license activation.

## Paywall behavior matrix

| state.kind            | Page UI                                            | Banner                       |
| --------------------- | -------------------------------------------------- | ---------------------------- |
| `trial-active-quiet`  | normal route                                       | none                         |
| `trial-active-day50`  | normal route                                       | "10 days left in your Aria trial." (info) |
| `trial-active-day55`  | normal route                                       | "5 days left." (warn)        |
| `trial-active-day59`  | normal route                                       | "N days left. Trial ends …." (urgent) |
| `trial-expired-grace` | normal route                                       | "Your trial has ended …." (urgent) |
| `trial-locked`        | PaywallScreen ("Your trial has ended")             | none                         |
| `pro-active`          | normal route                                       | none                         |
| `pro-grace`           | normal route                                       | none (Settings badge only)   |
| `pro-locked`          | PaywallScreen ("We couldn't verify your sub")      | none                         |
| `clock-skew-warn`     | wraps base — page renders normally if base allows  | "Your system clock looks off." (info, no Subscribe) |

## Read-only allow-list (locked-state)

These routes render their normal content even when `trial-locked` / `pro-locked`. Anything not in the list renders `<PaywallScreen/>`.

```
/settings   /briefing   /transcripts   /meetings   /approvals   /inbox
/calendar   /insights   /recap         /learning   /routing-log /tasks   /ask
```

Adding a new write-action route requires NO code edit — the guard covers it by default. Adding a new read-only-safe route is a one-line append in `READ_ONLY_ALLOW_LIST` at `src/renderer/app/routes.tsx`.

### E2E skip rationale

Both E2E specs document explicit prerequisites:
1. **Test public-key override** — `ENTITLEMENT_PUBLIC_KEY_HEX` is currently a compile-time embedded constant. A test mode that accepts JWTs from a separate test seed is needed.
2. **`__e2eSeedEntitlement` IPC hook** — there is no test-only IPC to seed the singleton `entitlement` row directly with a chosen state (analogous to `__e2eSeedReady` for approvals).
3. **Mock LICENSE_SERVER_URL** — needs a Playwright-bound HTTP server fixture wired into the activate / refresh / portal endpoints.

These are operator-gated additions and would themselves be a separate plan. The underlying invariants are fully covered at the unit + static layers (LockedGuard + 5-surface chokepoint + deep-link parser + handler unit tests).

## Live UAT script

Operator should walk these 5 scenarios before declaring Phase 08.1 done. Each requires the deployed Cloudflare Worker + Stripe Checkout link + email-restore wiring from Plan 01.

1. **Trial start.** Fresh install, no `entitlement` row. App should hit `/v1/trial/start`, get a signed JWT, write the row. Settings → Subscription shows "Trial · 60 days left". No banner visible.

2. **Day-59 banner.** Set system clock to 59 days after trial start. Settings still works, banner reads "1 day left. Your trial ends <date>." in urgent tone (red). Subscribe button opens Stripe Checkout in the system browser.

3. **Trial-locked paywall + restore via paste.** Set system clock to 61 days after trial start. Every write-action route renders PaywallScreen. `/settings` + `/briefing` still reachable. Click "I have a license key", paste a valid ARIA-… key, click Activate. UI transitions out of paywall within 1 frame of the push event.

4. **Trial-locked paywall + restore via deep link.** Same setup as scenario 3. Click an `aria://activate?key=ARIA-…` link from the email. The renderer's PaywallScreen disappears within 2s (push event emitted by main, no manual refresh).

5. **Customer Portal open + cancellation.** From Settings → Subscription on a pro-active row, click "Manage subscription". Stripe Customer Portal opens in browser. Cancel in the portal. Click "Refresh now" — within ~1 day of last-verified, the row stays `pro-active`; after 14 days of failed refreshes the state machine transitions to `pro-locked` and the next page nav shows the "We couldn't verify your subscription" PaywallScreen.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] EntitlementProvider was overwriting test-supplied `initialState` prop.**
- **Found during:** Task 2 PaywallScreen test pass.
- **Issue:** The provider always ran `entitlementGetState()` on mount and overwrote whatever was in state. Tests passing `initialState={PRO_LOCKED}` still ended up with whatever the mocked IPC returned (`TRIAL_LOCKED`), causing "pro-locked" assertions to fail.
- **Fix:** Pin `skipInitialFetchRef.current = props.initialState != null` and short-circuit the initial-fetch effect when set. Production paths never pass `initialState`, so behavior is identical.
- **Files modified:** src/renderer/features/entitlement/EntitlementProvider.tsx
- **Commit:** b680ca7

**2. [Rule 3 — Blocker / latent 08.1-02 gap] Deep-link activation did not emit ENTITLEMENT_STATE_CHANGED.**
- **Found during:** Task 9 — drafting the activate-via-deep-link E2E.
- **Issue:** `src/main/index.ts` called `handleActivateDeepLink(url, { service: entitlementService })` without an `emitStateChanged` callback. Successful activation updated the DB row but the renderer's paywall did NOT refresh until the next `ENTITLEMENT_GET_STATE` poll — defeating the "1-frame transition" UX invariant for the deep-link path.
- **Fix:** Wire `emitStateChanged` that looks up the live BrowserWindow at call time (mainWindow is constructed AFTER the `__ariaOnAriaUrl` forwarder is registered, so we cannot capture it in closure) and pushes the same `ENTITLEMENT_STATE_CHANGED` event the activate IPC handler uses.
- **Files modified:** src/main/index.ts
- **Commit:** 6320dc3

**3. [Rule 1 — Bug] Unused `userEvent` import in ActivateLicenseForm.test.tsx surfaced by typecheck.**
- **Found during:** post-Task-6 typecheck.
- **Fix:** Drop the import. Tests use `fireEvent` only.
- **Commit:** 57c6e3f
