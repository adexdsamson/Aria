<constraints>
- Use migration filename/version `123_meeting_notes.sql`. Do not use `015_*`; the runner would skip it after `012a` user_version 121.
- Raw transcript text must never be written to logs.
- No bot/recording SDKs, no live meeting join APIs, no cloud recording endpoints.
- Supported v1 input formats in this plan: paste text, `.txt`, `.vtt`, `.srt`, and best-effort `.json`. Defer `.docx` and `.pdf` unless dependencies already exist and tests are cheap.
</constraints>

<task type="auto" tdd="true">
  <name>Task 1: Migration 123 note/transcript schema</name>
  <behavior>
    - Creates `meeting_note`, `meeting_note_segment`, and indexes.
    - `meeting_note.normalized_text` stores full transcript text in SQLCipher.
    - `event_provider_key`, `event_account_id`, and `calendar_event_id` are nullable for standalone notes.
    - Segment rows store `start_offset`, `end_offset`, optional `speaker`, optional `timestamp_sec`.
    - `source_kind` is limited to `paste`, `txt`, `vtt`, `srt`, `json`.
  </behavior>
  <verify>
    <automated>node node_modules/vitest/vitest.mjs run tests/unit/main/db/migrations-123.spec.ts --reporter=dot</automated>
  </verify>
</task>

<task type="auto" tdd="true">
  <name>Task 2: Transcript normalization boundary</name>
  <behavior>
    - `normalizeTranscript(input)` returns `{ normalizedText, segments }`.
    - Raw text creates one full-span segment.
    - VTT/SRT preserve cue text and timestamp seconds.
    - JSON best-effort parser accepts common transcript arrays/objects with text/speaker/start/end.
    - Offsets are valid: `0 <= start < end <= normalizedText.length`.
  </behavior>
  <verify>
    <automated>node node_modules/vitest/vitest.mjs run tests/unit/main/transcripts/normalize.test.ts --reporter=dot</automated>
  </verify>
</task>

<task type="auto" tdd="true">
  <name>Task 3: Ingest service + calendar link heuristic</name>
  <behavior>
    - `ingestTranscriptNote(db, input)` is the sole write path for notes.
    - Links to nearest calendar event within +/-60 minutes when confidence is high.
    - If confidence is low, stores note standalone and returns top candidates for UI picker.
    - Fuzzy scoring considers event title, attendee names/emails, transcript title, and upload time.
  </behavior>
  <verify>
    <automated>node node_modules/vitest/vitest.mjs run tests/unit/main/transcripts/ingest.test.ts tests/unit/main/transcripts/link-calendar.test.ts --reporter=dot</automated>
  </verify>
</task>
